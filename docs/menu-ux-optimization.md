# 菜单/子菜单优化建议清单

> 目标：在不改变功能逻辑的前提下，统一命名、减少理解成本、提高首次使用成功率。

## 1. 命名规范（建议长期执行）

- 统一使用“功能名 + 结果导向”命名，避免隐喻词。
- 主菜单优先使用名词短语：`系统状态`、`模型配置`、`消息渠道`、`安全设置`、`Skills 管理`、`连通性测试`、`高级设置`、`服务管理`。
- 子菜单动词统一：`查看`、`刷新`、`安装`、`更新`、`修复`、`删除`、`导出`、`恢复`。
- 保持同一层级长度接近，避免一屏内出现超长说明。

## 2. 主菜单优化建议

1. `系统状态`
- 建议保留：安装状态、Gateway 状态、运行指标、配置文件存在性。
- 建议新增文案：在标题下固定显示“只读，不改配置”。

2. `模型配置（AI）`
- 建议保留：官方向导入口 + 状态刷新。
- 建议新增文案：说明“仅调用官方 `openclaw onboard`，本菜单不覆盖自定义模型路由”。

3. `消息渠道插件（官方）`
- 建议保留：官方渠道管理入口。
- 建议新增文案：社区渠道默认不在主流程安装，减少故障面。

4. `安全设置`
- 建议将开关名统一为可理解表达：
  - `系统命令执行`
  - `文件访问`
  - `网络访问`
  - `沙盒模式`
  - `启动记忆（Boot-MD）`
  - `会话记忆（Session-Memory）`

5. `Skills 管理`
- 建议保留三级能力：默认包、增强包、高级包。
- 建议把“安装概览”固定在该菜单顶端，减少来回切换。

6. `连通性测试`
- 建议将“API 测试/渠道测试/诊断命令”分组显示（已分组，可继续保持）。
- 建议在每个测试项后给出失败后下一步动作（如 `doctor --fix`）。

7. `高级设置`
- 建议按风险分区：
  - 低风险：升级、备份、日志
  - 中风险：迁移、修复
  - 高风险：重置、卸载

8. `配置修复与迁移`
- 建议明确“保留用户会话与记忆数据”的默认行为（已具备能力，建议强化提示）。

## 3. 子菜单文案优化建议

1. 用语一致性
- 将“总览/概览”统一成一个词，推荐使用“概览”。
- 将“刷新状态”统一成“刷新运行状态”或“刷新配置状态”，避免歧义。

2. 输出风格一致性
- 信息输出顺序建议固定：
  - 结论（成功/失败）
  - 影响范围
  - 下一步命令

3. 错误提示模板
- 建议统一为：
  - `问题描述` → `可能原因` → `修复命令` → `文档链接`

## 4. 信息架构优化建议（不改功能）

1. 减少跨菜单跳转
- 在 `系统状态`、`连通性测试` 结果页直接提示对应菜单入口编号。

2. 把“高频项”放前面
- 高访问路径建议：`模型配置` → `消息渠道` → `连通性测试`。

3. 减少“占位”字样暴露
- 对 V1 占位页面建议标注“预览功能（只读）”，比“占位”更工程化。

## 5. 实施优先级

1. P0（立即）：统一主菜单与高频子菜单命名。
2. P1（本周）：统一错误提示模板与测试失败后的下一步动作。
3. P2（后续）：补全每个子菜单顶部的“操作影响说明”。
